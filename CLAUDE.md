# Knot

Knot is a functional relational programming language. Relations are the primary data structure, computation is pure and functional, and state is automatically persisted to SQLite.

See `DESIGN.md` for the full language specification.

## Project Structure

Cargo workspace with three crates:

```
crates/
  knot/              Frontend library (lexer, parser, AST, diagnostics)
  knot-runtime/      Rust staticlib linked into compiled programs (value representation, SQLite persistence)
  knot-compiler/     Cranelift-based compiler producing native executables (CLI binary: knotc)
examples/            Example .knot programs
```

## Build & Test

```sh
cargo build                  # Build all crates
cargo test                   # Run all tests (268 parser/lexer + 134 compiler tests)
cargo test -p knot           # Run only frontend tests
```

## Compiling Knot Programs

```sh
cargo run -p knot-compiler -- build examples/hello.knot
./examples/hello
```

The compiler (`knotc`) looks for `libknot_runtime.a` next to its own executable. In a cargo workspace, both end up in `target/<profile>/`, so this works automatically. Override with `KNOT_RUNTIME_LIB=/path/to/libknot_runtime.a`.

Compiled binaries create a `knot.db` SQLite database in the current directory for persistence.

## Architecture

### Compilation Pipeline

```
source.knot → Lexer → Tokens → Parser → AST → Desugar → Type Inference → Effect Inference → Type Resolution → Cranelift IR → .o → cc link → executable
```

### Frontend (`crates/knot/`)

- **Lexer** (`lexer.rs`): Tokenizer with layout-sensitive newline handling
- **Parser** (`parser.rs`): Recursive-descent with Pratt expression parsing for operator precedence; `do` blocks are parsed in atom position so they can appear directly as function arguments (e.g. `f do ...` or `f (do ...)`); postfix type annotations supported on any expression (`expr : Type`) with and without surrounding parens — `parse_expr` greedily consumes a trailing `:` and parses a type, wrapping the result in `ExprKind::Annot`; `%` lexes as `TokenKind::Percent` and parses into `BinOp::Mod`
- **AST** (`ast.rs`): Spanned AST nodes for all language constructs
- **Diagnostics** (`diagnostic.rs`): Error reporting with source locations

Public API:
```rust
let lexer = knot::lexer::Lexer::new(&source);
let (tokens, lex_diags) = lexer.tokenize();
let parser = knot::parser::Parser::new(source, tokens);
let (module, parse_diags) = parser.parse_module();
```

### Runtime (`crates/knot-runtime/`)

Compiled as a `staticlib` (with `rlib` for workspace dependency resolution). All functions use `extern "C"` ABI, called by Cranelift-generated code via symbol references.

- All Knot values are `*mut Value` (heap-allocated tagged enum); integers are unbounded (`num_bigint::BigInt`); IO actions are `Value::IO(fn_ptr, env)` thunks executed via `knot_io_run`
- `rusqlite` with `bundled` feature statically links SQLite into every compiled binary
- Source relations map to SQLite tables with schema derived from Knot types
- Schema descriptor format for records: `"name:text,age:int"` (passed as string constants from generated code)
- Schema descriptor format for direct ADT relations: `"#Circle:radius=float|Rect:width=float;height=float"` — `#` prefix signals ADT schema, `|` separates constructors, `:` separates name from fields, `;` separates fields, `=` separates field name from type; runtime creates wide table with `_tag TEXT` + all constructor fields as nullable columns
- Column type `tag` for enum-like ADT fields (all nullary constructors): stored as TEXT in SQLite, reconstructed as `Constructor(tag, Unit)` on read
- Automatic indexing: the runtime observes query patterns and creates indexes lazily — ADT tables get a `_tag` index at init; columns in `DELETE WHERE`, `UPDATE WHERE`, and `READ WHERE` clauses are auto-indexed on first use via `ensure_indexes_for_where(table, where_clause)`, which extracts unqualified `"col"` identifiers; columns referenced in the `WHERE`/`ORDER BY` clauses of pushed-down SELECT/aggregate queries (`knot_source_query`, `knot_source_query_count`, `knot_source_query_sum`, `knot_source_query_float`, `knot_source_query_value`) are auto-indexed via `ensure_indexes_for_sql(sql)` which parses `FROM "_knot_<name>" [AS] <alias>` to build an alias→table map, then attributes qualified `<alias>."col"` refs to the right table and unqualified `"col"` refs to the single `_knot_` table when the query touches only one (multi-table unqualified refs are skipped as ambiguous); covers filtered aggregates (`countWhere`, filtered `minOn`/`maxOn`/`sum`/`avg`), `sortBy` ORDER BY targets, and multi-table join keys (`where e.dept == d.name` indexes both `_knot_employees.dept` and `_knot_departments.name`); `KnotDb` tracks created indexes in a per-session `HashSet` to avoid redundant DDL; uses `CREATE INDEX IF NOT EXISTS` for cross-session idempotency
- Nested relation fields (`[T]` inside records): stored in **child tables** (`_knot_{source}__{field}`) with `_parent_id` FK linking to parent's `_id`; parent tables with nested children get `_id INTEGER PRIMARY KEY AUTOINCREMENT`; schema descriptor format `field:[child_schema]` (bracket-aware parsing via `split_respecting_brackets`); deeply nested relations recurse arbitrarily
- Concurrency: `DB_PATH: Mutex<String>` stores the DB path globally so spawned threads can open their own connections; `THREAD_HANDLES: Mutex<Vec<JoinHandle<()>>>` tracks spawned threads; `ACTIVE_FORKS: AtomicUsize` + `ACTIVE_FORKS_MUTEX`/`ACTIVE_FORKS_CVAR` track all background threads (used so `race` losers that outlive the parent still get joined at program exit); `knot_fork_io` deep-clones the IO value's environment (recursive `deep_clone_value` using `Box::new`, not arena), spawns an OS thread with its own `KnotDb` connection, stores the JoinHandle; `knot_threads_join` drains and joins all handles and also waits on `ACTIVE_FORKS_CVAR` until detached race losers settle; each spawned thread gets its own arena allocator automatically
- `race` primitive: `knot_race_io` spawns two workers, each with its own `KnotDb`, deep-cloned IO value, and a per-thread `CancelToken` (`AtomicBool` flag + `(Mutex<()>, Condvar)` notify pair); the winner is the first worker to complete and acquire the outcome `Mutex<Option<(bool, *mut Value)>>`; on winning, it signals the loser's cancel token (`peer_cancel.cancel()`) and notifies the outcome `Condvar`; the parent waits only for the winner — the loser is not joined here but is tracked via `ACTIVE_FORKS` so program-exit join still waits for it; cooperative cancellation: `cancel_requested()` (reads thread-local `CANCEL_FLAG: RefCell<Option<Arc<CancelToken>>>`) is polled by `knot_io_run` between every thunk; `knot_sleep` uses `wait_for_cancel` (parks on the condvar with `wait_timeout_while`) when a CancelToken is installed so a sleeping loser wakes immediately on cancel; the winning value is deep-cloned into Box storage (not arena) before being wrapped in `Constructor("Err"/"Ok", ...)` so it outlives the worker thread's arena; `race` is in `CANNOT_ROLLBACK_BUILTINS` (cannot appear inside `atomic`)
- STM retry: `RELATION_CHANGED: (Mutex<u64>, Condvar)` — global change counter incremented by `notify_relation_changed()` on every relation write (`set`, `replace`, plus row-level UPDATE/DELETE paths); `knot_stm_retry()` sets a thread-local `STM_RETRY` flag; `knot_stm_snapshot()` reads the current counter; `knot_stm_wait(snapshot)` blocks on the Condvar until counter exceeds snapshot **AND** at least one write event since the snapshot matches the watcher's read filters; `knot_stm_check_and_clear()` checks and resets the thread-local retry flag; `knot_atomic_rollback(db)` rolls back the current savepoint
- STM row-level invalidation: each table-read inside `atomic` registers a `ReadFilter` in a thread-local read set. `ReadFilter::All` = "wakes on any write to this table"; `ReadFilter::Cols(Vec<(String, ColPred)>)` = AND-conjunction of column predicates (multiple `Cols` filters per table OR together at slot level). `ColPred::Cmp(CmpOp, SqlVal)` supports `Eq`/`Neq`/`Lt`/`Le`/`Gt`/`Ge`; `ColPred::In(Vec<SqlVal>)` supports `col IN (v1, v2, ...)`. Codegen detects `single (filter (\r -> r.col OP expr) rows)` patterns and SQL-pushed-down query patterns: emits the broad `knot_stm_track_read` first (so the table is on the read set), then `knot_stm_track_read_pred(name, spec, params_rel)` to downgrade the most-recent `All` filter into a precise `Cols`. The spec string format is `col:op:idx[,idx,...];...` joined by `;` (one predicate per segment): `op` ∈ `=`, `!=`, `<`, `<=`, `>`, `>=`, `in`; indices reference rows in `params_rel` (the same relation passed to `knot_source_query`). Parse errors or out-of-range indices leave the broad `All` filter in place (safe fallback). Writes emit `WriteEvent::Bulk` (wakes everyone) or `WriteEvent::Rows(Vec<HashMap<String, SqlVal>>)` (per-row column payloads); `WriteEvent::matches_filters` walks each `ReadFilter` and calls `row_matches_preds`, which evaluates each `(col, pred)` against the row using `sql_partial_cmp` for cross-type comparisons (Int↔Real, Int↔Text for BigInt-as-TEXT). Missing payload columns or incompatible types fall back to "wake conservatively" (return true). UPDATE and DELETE paths now emit per-row `Rows` events (not just bulk), so STM watchers can filter on those writes too.

### Compiler (`crates/knot-compiler/`)

- **Desugaring** (`desugar.rs`): Transforms "pure comprehension" do blocks into nested calls to `__bind`, `__yield`, and `__empty`. A pure comprehension is a do block with only Bind/Where/Let/Yield statements (no bare side-effecting expressions or `groupBy`). Do blocks that are direct values of `set`/`replace` expressions are NOT desugared (preserves SQL optimization patterns). Do blocks containing `groupBy` are NOT desugared (require loop-based codegen for the collect/group/re-iterate pattern). Do blocks containing constructor pattern binds (`Ctor pat <- expr`) are NOT desugared (the desugarer cannot syntactically distinguish value pattern matches from monadic binds; direct codegen handles both). Do blocks containing IO-returning expressions are NOT desugared — the desugarer's `expr_is_io` recognizes both external IO builtins (`println`, `readFile`, `now`, etc.) and relation operations (`SourceRef`, `DerivedRef`, `Set`, `ReplaceSet`, `At`) as IO, excluding these do blocks from desugaring so they go through the dedicated IO codegen path (`compile_io_do`) which runs each IO action with `knot_io_run` and wraps the result in `knot_io_pure`; this ensures IO do blocks inside if/then/else branches are typed correctly as IO rather than defaulting to relational. Mixed/sequential do blocks remain as `Do(stmts)` nodes for imperative codegen. The desugared `__bind`/`__yield`/`__empty` dispatch through Monad/Applicative/Alternative trait impls — the monad type is resolved at compile time via HKT unification (`monad_info` maps spans to `MonadKind::Relation`, `MonadKind::Adt(name)`, or `MonadKind::IO`), and codegen calls the appropriate `Monad_{type}_bind`, `Applicative_{type}_yield`, or `Alternative_{type}_empty` mangled function. This allows `do` blocks to work with any type implementing the Monad/Applicative/Alternative traits, not just `[]`.
- **Type inference** (`infer.rs`): Hindley-Milner type inference with row-polymorphic records and row-polymorphic variants, let-generalization, unification with occurs check, ADT constructor typing, trait method registration, associated type erasure, higher-kinded type polymorphism (`TyCon`/`App` type nodes with normalization-based unification), IO monad type (`Ty::IO(BTreeSet<IoEffect>, Box<Ty>)` with effect tracking for `Console`, `Fs`, `Network`, `Clock`, `Random`), units of measure (`Ty::IntUnit(UnitTy)`/`Ty::FloatUnit(UnitTy)` with normalized exponent-map representation `BTreeMap<String, i32>` plus optional `UnitVar` for polymorphism; unit algebra: `+`/`-` require unit match, `*`/`/` compose units, negation preserves units; plain `Int`/`Float` are unit-agnostic and unify with any `IntUnit`/`FloatUnit`; `Scheme` quantifies over both `TyVar`s and `UnitVar`s — instantiation freshens both; all stdlib functions with numeric types are unit-polymorphic — `avg` registered as `∀a u. (a -> Float<u>) -> [a] -> Float<u>`; `sum` already fully polymorphic in return type; `randomInt` as `∀u. Int<u> -> IO {random} Int<u>` (preserves bound unit in result); `randomFloat` as `∀u. IO {random} Float<u>`; `now` as `IO {clock} Int<Ms>`; `sleep` as `Int<Ms> -> IO {clock} {}`; `Ms` registered as built-in unit (milliseconds) for clock operations; `count` as `∀a u. [a] -> Int<u>`; `length` as `∀u. Text -> Int<u>`; `take`/`drop` as `∀u. Int<u> -> Text -> Text`; `listen` as `∀a b u. Int<u> -> (a -> b) -> IO {network} {}`; `bytesLength` as `∀u. Bytes -> Int<u>`; `bytesSlice` as `∀u1 u2. Int<u1> -> Int<u2> -> Bytes -> Bytes`; `bytesGet` as `∀u1 u2. Int<u1> -> Bytes -> Int<u2>`; unit declarations collected in `collect_types`; derived unit aliases expand at use site), refined types (`type Nat = Int where \x -> x >= 0` — refined type aliases stay nominal as `Ty::Con(name, vec![])`, NOT resolved to base type; stored in `refined_types: HashMap<String, (Ty, Expr)>` mapping name to base type and predicate; subsumption rule in `unify`: when `Con(name, [])` is a refined type, unify with its base type — enables bidirectional compatibility `Nat ↔ Int`; `refine expr` typed as `Result RefinementError alpha` where `alpha` is fresh and constrained by context — post-inference resolves `alpha` to the refined type name for codegen; `RefinementError` registered as a type alias to `{typeName: Text, violations: [{field: Maybe Text, message: Text}]}`; `TypeKind::Refined { base, predicate }` in AST for inline refinements, stripped to base during schema resolution; `refine_vars: Vec<(Span, TyVar, Ty)>` tracks refine expressions for post-inference target resolution; `RefineTargets` and `RefinedTypeInfoMap` exported to codegen), and trait bound checking (explicit `TraitName a =>` constraints on function signatures are validated at call sites against known `impl` declarations; trait method calls automatically carry their trait's constraint). All relation operations return IO: `SourceRef` returns `IO {} [T]`, `DerivedRef` returns `IO {} [T]`, `Set`/`ReplaceSet` return `IO {} {}`; `Set`/`ReplaceSet` unify target and value by unwrapping IO from both sides. `Atomic` is typed as `IO {} a -> IO {} a` — body must be IO, enforced via unification; `in_atomic: bool` flag tracks context for `retry` validation. Built-in registrations include `fork : ∀r. IO r {} -> IO {} {}` (effect row of the spawned IO is discarded; the caller only sees `IO {} {}`), `race : ∀a b r. IO r a -> IO r b -> IO r (Result a b)` (both args share an effect row that propagates to the result), `randomUuid : IO {random} Uuid`, and `retry : ∀a. a` (bottom type). New primitive `Ty::Uuid` in `infer.rs` with parser/printer/erasure plumbing parallel to `Ty::Bytes`. Built-in ADTs registered in `register_builtins`: `Maybe a` (`Nothing`/`Just`), `Result e a` (`Err`/`Ok`), `Bool` (`True`/`False`) — `Bool` constructors map to the primitive `Ty::Bool` (not `Ty::Con`), so `True {}` and `False {}` produce the same type as `true`/`false` literals; `Result` has built-in Functor/Applicative/Monad/Alternative impls for do-block support. Constructor patterns in `case` and do-bind create open variant types `<Ctor {} | r>` (the scrutinee only needs to *contain* the matched constructor), enabling row-polymorphic variant matching; when all constructors of a known data type are covered without a wildcard, the row variable is closed and exhaustiveness is checked normally; open variants require a wildcard `_` case
- **Effect inference** (`effects.rs`): Infers per-declaration effects (`{r *rel}`, `{w *rel}`, `{rw *rel}` for both, `{console}`, `{clock}`, etc.), checks IO-in-atomic constraints (atomic blocks must only contain DB interactions — no console/fs/clock/random), validates that atomic blocks actually interact with relations (have reads or writes), validates explicit effect annotations against inferred effects. `rw *x` is shorthand for `r *x, w *x` — parsed as both effects, and the display coalesces matching pairs back to `rw *x`.
- **Type resolution** (`types.rs`): Resolves aliases (including multi-variant ADTs to `ResolvedType::Adt`), computes SQLite schemas from Knot type annotations, collects subset constraints; refined type aliases separated during collection — predicates stored in `refined_types: HashMap<String, Expr>`, base types stored in `aliases` for schema resolution; `TypeKind::Refined` transparent for schema — `resolve_type` and `apply_type_subst` recurse into base; `source_refinements: HashMap<String, Vec<(Option<String>, String, Expr)>>` maps source names to their field-level and whole-element refinement predicates (collected by `collect_source_refinements` walking source type ASTs)
- **Codegen** (`codegen.rs`): Cranelift IR generation — the `build_function` pattern moves `ctx`/`builder_ctx` out of `self` to avoid borrow conflicts while allowing `self.method()` calls during IR building
- **Linker** (`linker.rs`): Invokes `cc` with platform-appropriate flags
- **CLI** (`main.rs`): `knotc build <file.knot>` entry point

Key codegen patterns:
- All values are pointer-typed (`ptr_type`) in Cranelift IR — uniform representation
- `do` blocks compile to nested loops with SSA block params for iteration counters
- `where` clauses become conditional branches; skip blocks jump to the loop's continue block
- Functions are constants bound to lambdas (`add = \x y -> x + y`); when a Fun's body is a Lambda, the compiler extracts its params for direct Cranelift function compilation (no closure overhead)
- Standalone lambdas compile as separate functions; free variables captured in a record-valued closure environment; multi-param lambdas (`\a b c -> body`) are curried into nested single-param lambdas at compile time
- Runtime functions are pre-declared as imports; `call_rt`/`call_rt_void` helpers emit calls
- `knot_relation_len` returns raw `usize`, not a boxed `Value` — use directly as loop bound
- Trait impl methods compile as mangled functions (`TraitName_TypeName_methodName`); a dispatcher function checks `knot_value_get_tag` at runtime and calls the matching impl; missing impls panic with a clear error message (except operator-mapped methods like `eq`/`compare`/`add`/`sub`/`mul`/`div`/`negate` which fall back to runtime functions for types without explicit impls)
- Operator trait dispatch: arithmetic operators (`+`, `-`, `*`, `/`, `%`) dispatch through `Num` trait methods (`add`, `sub`, `mul`, `div`, `mod`); unary negation dispatches through `Num.negate`; `==` dispatches through `Eq.eq`; `!=` dispatches through `Eq.eq` then negates; comparison operators (`<`, `>`, `<=`, `>=`) dispatch through `Ord.compare` and check the resulting `Ordering` constructor tag; `&&`/`||`/`++` remain direct runtime calls (no trait). Primitive impls (Int, Float, Text, Bool) are registered as intrinsic codegen impls that delegate to runtime functions (`knot_value_mod` for `%`, etc.), avoiding circular dependencies (the prelude source does NOT contain these impls). `%` SQL-pushes down to SQLite's `%` operator inside comprehensions where the SQL lint pass already permits the surrounding expression.
- Default trait methods: if an impl omits a method with a default body, the default is auto-compiled for that type
- `deriving (TraitName)` on data types auto-generates impls using the trait's default method bodies
- `show` calls `knot_value_show` (converts any value to Text representation)
- Standard library functions are registered as 1-param user_fns with currying: 1-param functions delegate directly to runtime; 2-param functions (filter, match, map, take, drop, contains) curry via `define_stdlib_fn_2` — outer function captures first arg in a closure env, inner closure extracts it and calls the runtime; 3-param functions (fold) use double currying via `define_stdlib_fn_3`
- Recursive derived relations: detected via `expr_contains_derived_ref` AST walk; compile to a body function `knot_user_<name>_body(db, self_val)` where self-references read from `self_val` param (via `__derived_self_<name>` env key), plus a wrapper that calls `knot_relation_fixpoint(db, body_fn_ptr, empty)` for fixed-point iteration
- Bool constructors: `True` and `False` are special-cased in codegen — bare `True`/`False` and applied `True {}`/`False {}` compile to `knot_value_bool(1)`/`knot_value_bool(0)` (not `knot_value_constructor`); pattern matching on `True`/`False` in `case` uses `knot_value_get_bool` instead of tag string comparison
- IO do-blocks: detected via `is_io_do_block` (checks if any statement binds from or is a bare IO-returning expression — includes both external IO builtins and relation operations like SourceRef/DerivedRef/Set/ReplaceSet/At); compiled by `compile_io_do_as_thunk` which runs each IO action inline with `knot_io_run(db, io)` and wraps the final result in `knot_io_pure`; `compile_main` calls `knot_io_run(db, result)` on the main function's return value so IO main functions execute their effects; after IO run, `knot_threads_join()` is called to wait for all spawned threads before db close
- Concurrency codegen: `fork` registered as 1-param stdlib function via `define_stdlib_fn_1("fork", "knot_fork_io")`; `Atomic(inner)` compiles to a retry loop with three Cranelift blocks (loop_block, retry_block, done_block) — loop_block snapshots the change counter via `knot_stm_snapshot`, begins savepoint, runs body, checks `knot_stm_check_and_clear` retry flag with `brif`; retry_block calls `knot_atomic_rollback` then `knot_stm_wait(snapshot)` then jumps back to loop_block; done_block commits and continues; `Var("retry")` compiles to `knot_stm_retry()` call; all blocks explicitly sealed after predecessors are emitted
- IO runtime representation: `Value::IO(fn_ptr, env)` where fn_ptr has signature `extern "C" fn(db, env) -> *mut Value`; `knot_io_pure` wraps a value in a trivial IO (null fn_ptr, value as env); `knot_io_run` executes thunks or returns non-IO values as-is; `knot_io_bind` and `knot_io_then` compose IO actions via nested thunks with captured environments
- Effectful builtins (`println`, `readFile`, `now`, etc.) compile to `_io` wrapper functions (e.g. `knot_println_io`) that capture arguments and return `Value::IO` thunks instead of executing immediately
- HTTP client (`fetch`/`fetchWith`): compiler-special-cased forms that reuse route declarations for type safety; `compile_fetch` extracts the constructor's route entry metadata (method, path pattern, body/query/request-header field descriptors, response schema, response-header descriptors) at compile time and passes them as string constants alongside the compiled record payload to `knot_http_fetch_io`, which returns an IO thunk; the thunk builds the URL (substituting path params), serializes body fields to JSON, extracts route-declared request headers from payload fields (converting camelCase names to HTTP-Header-Case via `camel_to_header_case`), sets ad-hoc headers from `fetchWith` options, executes the HTTP request via `ureq`, parses the JSON response, and wraps it in a `Result` ADT (`Ok {value: parsed}` or `Err {error: {status, message}}`); when response headers are declared, the Ok value is wrapped as `{body: parsed, headers: {h1: v1, ...}}` with response headers parsed from the HTTP response; type inference (`try_infer_fetch`) fully handles these forms — it resolves the response type from `fetch_response_types` and wraps the response type with `{body: T, headers: H}` when `fetch_response_headers` are present (both populated from route declarations during `pre_register`)
- `serve` expression: `serve API where E1 = h1; E2 = h2; ...` produces a value of type `Server API _` (a row variable when handlers have no concrete effects) or `Server API {effects}` when handlers do — e.g. `Server API {console}` if a handler calls `println`. Internally `Server` is `Ty::Con("Server", [api_ty, effect_row])`. The effect row uses the same display rules as `IO`: an unresolved row variable renders as `_` (parser shorthand for `{| _}`), `{}` for closed empty, `{e1, e2}` or `{e1 | _}` when concrete. Type inference (`infer_serve`) enforces exhaustiveness — every endpoint constructor of the api must have exactly one handler, no duplicates, no extras. Each handler is checked against the type `input_record -> IO {| r} (Result HttpError response_ty)` derived from its route entry (input record built from path/query/body/headers fields, response type from the route's `->` declaration; effect row is open to allow IO-returning handlers; the response is wrapped in `Result HttpError` so handlers can return `Ok {value: T}` for 200 or `Err {error: {status, message}}` for custom HTTP status codes). `HttpError = {status: Int, message: Text}` is registered as a built-in type alias in `register_builtins` next to `RefinementError`. `RequestCtx = {clientIp: Text, receivedAt: Int<Ms>, header: Text -> Maybe Text}` is registered alongside them and is the parameter type of a route's `rateLimit.key` function. Codegen (`compile_serve`) desugars to `\req -> case req of Ctor pat -> handler pat` reusing the existing case+lambda paths — no special runtime representation, since all values are uniform `*mut Value` so heterogeneous response types are fine. `listen : Int<u> -> Server a r -> IO {network | r} {}` takes the server value directly and unifies its row variable with the server's effect row, so handler effects (e.g. `console`) propagate to `listen`'s IO type; the runtime listen loop dispatches each request to the corresponding handler, then inspects the returned `Result` constructor: `Ok` → status 200 with `value` serialized as JSON, `Err` → status from `error.status` (clamped to 100..=599) with `{error: error.message}` as the JSON body. When response headers are declared, the `Ok` value is `{body: T, headers: H}` and headers flow into the response.
- `groupBy` in do-blocks: compiles as a two-phase loop — phase 1 collects qualifying rows into a temp relation (inside existing loops), closes those loops, then calls `knot_relation_group_by(db, temp, schema, key_cols)` which uses SQLite for grouping: inserts row indices + key column values into a temp table, executes `ORDER BY` on key columns, and groups consecutive rows with matching keys in O(n); phase 2 opens a new loop over the groups and rebinds the primary bind variable to each group (a sub-relation); field access on groups delegates to the first element via `knot_record_field`'s `Value::Relation` handling; codegen extracts key column names from the record key expression and schema from `source_schemas`
- Refined types codegen: `compile_refine` looks up `refine_targets[span]` to find the refined type name, compiles the predicate lambda via `compile_expr`, calls the predicate on the value, branches to `build_ok_result` (constructs `Constructor("Ok", Record([("value", val)])`) or `build_refinement_err` (constructs `Constructor("Err", Record([("error", {typeName, violations})]))`); `emit_refinement_checks` inserts `knot_refinement_validate_relation` calls before `knot_source_write`/`knot_source_append` in `set` codegen; route refinement predicates compiled and registered via `knot_route_set_field_refinement` during init after `db` is created; Result monad dispatch: `compile_monadic_bind`/`yield`/`empty` special-case `type_name == "Result"` to call `knot_result_bind`/`knot_result_yield`/`knot_result_empty` runtime functions

## Supported Language Features
